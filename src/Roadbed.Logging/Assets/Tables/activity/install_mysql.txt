-- Roadbed.Logging :: activity (MySQL / MariaDB)

-- Run this script with the target database already selected:
--     CREATE DATABASE IF NOT EXISTS logging
--         DEFAULT CHARACTER SET utf8mb4
--         DEFAULT COLLATE utf8mb4_unicode_ci;
--     USE logging;
--     SOURCE install_mysql.txt;   -- or paste the contents into your workbench
--
-- The table is created unqualified; the host's LoggingOptions.Schema
-- should be set to the database name you selected (e.g. "logging") so
-- the C# repository qualifies every statement consistently.
--
-- Partitioning
-- ============
-- Partitioned monthly by RANGE on TO_DAYS(created_on) with 10 years of
-- pre-created partitions (2026-01 .. 2035-12 = 120 monthly partitions),
-- plus a p_min floor and a pmax catch-all. Retention is enforced
-- externally by dropping partitions older than the configured window;
-- the recommended window for `activity` is 12 months.
--
-- Partition key rules
--   * Every UNIQUE / PRIMARY key on a partitioned InnoDB table must
--     contain the partition column. Hence PRIMARY KEY (id, created_on).
--   * There is intentionally NO standalone UNIQUE (id) — uniqueness of
--     the caller-supplied UUIDv7 is guaranteed by the application.
--   * Partitioned InnoDB tables cannot have foreign keys; the lineage
--     references in `activity_input` are soft on purpose.
--   * Partition pruning only happens when the query filters on the
--     partition column (created_on). All composite indexes lead with the
--     fleet filter and end with created_on so the MCP can prune.
--
-- UTC contract
-- ============
-- All stored timestamps are UTC.
--   * created_on / recorded_on use DEFAULT (UTC_TIMESTAMP(6)) so server
--     time zone does not affect the partition / retention key.
--   * last_modified_on declares ON UPDATE CURRENT_TIMESTAMP(6) as a
--     safety net only; the Roadbed.Logging UPDATE path explicitly sets
--     last_modified_on = UTC_TIMESTAMP(6) on every statement so the
--     column is UTC regardless of the connection's session time_zone.
--   * Activity-id, trace, and span columns use ascii_bin so lexical
--     order equals chronological order. UUIDv7 (the standardized
--     identifier for activity ids) carries a big-endian millisecond
--     timestamp in its first 48 bits, so the hex string still sorts
--     chronologically under ascii_bin.
