-- Roadbed.DbQueue :: queue_message_{q} (MySQL / MariaDB)
--
-- Reference DDL template. Substitute the literal '{q}' below with the
-- queue's logical name (the same value passed to QueueDefinition<T>:
-- lowercase ASCII letters, digits, underscore; max 48 characters so
-- this table identifier fits inside MySQL's 64-character limit).
-- Then run this script with the business schema this queue lives in
-- already selected:
--
--     USE foo_subscriptions;          -- example
--     SOURCE install_mysql.txt;       -- or paste into your workbench
--
-- The table is created unqualified; the queue's IDataConnectionFactory
-- in C# must point at this schema.
--
-- Per spec §4 / D4, each queue lives IN the business schema it serves
-- (not a dedicated queue schema and not the logging schema). Multiple
-- queues in the same host may therefore live in different databases.
--
-- This is the IMMUTABLE message table: rows are INSERTed by EnqueueAsync
-- and never UPDATEd or DELETEd by the library. The companion
-- queue_processed_{q} table is INSERTed once per dispatch attempt
-- (success or failure) and is the basis for the app-level FIFO
-- anti-join in QueueProcessor.ProcessBatchAsync.
--
-- Partitioning
-- ============
-- Partitioned MONTHLY by RANGE on TO_DAYS(created_on) with 10 years of
-- pre-created partitions (2026-01 .. 2035-12 = 120 monthly partitions),
-- plus a p_min floor and a pmax catch-all. Add new monthly partitions
-- before pmax begins absorbing live rows; reorganizing pmax is far
-- more expensive than adding partitions ahead of it.
--
-- Retention is enforced EXTERNALLY by the DBA by dropping partitions
-- older than the configured window. The recommended default window is
-- 12 MONTHS (same as logging.activity). The DBA may override per queue
-- — a longer retention for a legal-record queue, a shorter retention
-- for a high-volume queue. See companion notes in the
-- queue_processed_{q} template for the mandatory DROP-PARTITION
-- ordering: message-table FIRST, processed-table SECOND. Dropping the
-- processed partition first orphans messages back into the claim pool
-- and re-runs the handler.
--
-- A message still unprocessed when its partition reaches the retention
-- boundary is SILENTLY DROPPED — the library cannot see this because
-- it does no retention. The host MUST alert on stale-unprocessed
-- backlogs before the boundary.
--
-- Partition key rules
--   * Every UNIQUE / PRIMARY key on a partitioned InnoDB table must
--     contain the partition column. Hence PRIMARY KEY (id, created_on)
--     and UNIQUE KEY (external_id, created_on).
--   * No foreign keys: partitioned InnoDB cannot carry FKs. The
--     reference from queue_processed_{q}.fk_queue_id back to id is a
--     LOGICAL reference, not an enforced constraint.
--   * The external_id uniqueness guarantee comes from C# (UUIDv7 via
--     Guid.CreateVersion7); the per-month composite UNIQUE here is a
--     defense-in-depth backstop within a single partition.
--
-- UTC contract
-- ============
-- created_on defaults to UTC via UTC_TIMESTAMP(6) so the partition key
-- never moves under the server's session time zone.
--
-- external_id is CHAR(36) ascii_bin: UUIDv7's first 48 bits are a
-- big-endian millisecond timestamp, so lexical order under ascii_bin
-- equals chronological order — useful for range scans on this column
-- even though the table's primary FIFO order is the auto-increment id.
